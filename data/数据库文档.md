采用“双轨制”（双库联动）架构**：**Milvus（向量数据库）负责语义检索和模糊匹配，**MySQL（关系型数据库）** 负责结构化数据的精确查询、过滤和前端展示。二者通过唯一的 `id` 进行强关联。

### 一、 向量检索表：`milvus_company`

**设计定位：** 专为大模型（LLM）和向量嵌入模型（如 BGE-M3）设计的检索表。这里的字段被极度精简，主要目的是**提高语义相似度搜索的命中率**，并支持最基础的标量过滤。

| **字段名称**               | **推测类型** | **业务含义 / 详细解释**         | **作用与核心价值**                                           |
| -------------------------- | ------------ | ------------------------------- | ------------------------------------------------------------ |
| **`id`**                   | String       | **唯一主键** (如 `COMP_000001`) | **联动主键**。当大模型通过语义向量匹配到这条数据时，通过此 ID 去 MySQL 中把该公司的完整档案拉取出来。 |
| **`core_text_for_bge_m3`** | Text         | **核心业务文本段**              | **这是 Milvus 的灵魂字段。** 它将公司的核心信息（如公司名称、经营范围）与**全量 JSON Metadata**拼装成了一段连贯的自然语言文本。专门喂给 BGE-M3 提取向量特征，应对模糊提问。 |
| **`source`**               | String       | **数据来源** (网址或来源标记)   | 用于在向量检索前做**标量前置过滤**（预过滤）。               |

###  关系型主表：`mysql_company`

**设计定位：** 企业的“全息数字档案”。用于存储经过清洗、去重、校验后的高价值结构化数据。当前端需要展示企业详情，或者业务系统需要精准过滤时，就会调用此表。

| **字段名称**                     | **推测类型** | **业务含义 / 详细解释**     | **核心质检与业务价值**                                       |
| -------------------------------- | ------------ | --------------------------- | ------------------------------------------------------------ |
| **`id`**                         | VARCHAR      | **唯一主键**                | 与 Milvus 库中的 ID 绝对一一对应。                           |
| **`source`**                     | VARCHAR      | **数据来源 URL / 站点**     | 数据溯源，如果数据出错可以追溯到原始网站。                   |
| **`company_name`**               | VARCHAR      | **企业名称**                | **核心底线字段**。企业实体标识，清洗时如果没有此字段会被一票否决。 |
| **`legal_rep`**                  | VARCHAR      | **法定代表人**              | 判断企业实体的真实性，排查关联交易的关键字段。               |
| **`est_date`**                   | DATE/STR     | **成立日期**                | 用于年限计算和过滤（如：成立满3年）。                        |
| **`capital`**                    | DECIMAL      | **注册资本 (纯数字)**       | 资金实力体现。**已通过清洗脚本绝对剥离非数字符号，并统一换算为以“元”为单位的纯数字**，方便做数学大小比对。 |
| **`company_type`**               | VARCHAR      | **企业类型**                | 例如：有限责任公司、国企、外资等，体现企业性质。             |
| **`reg_number`**                 | VARCHAR      | **工商注册号**              | 早期企业的唯一标识，现多与信用代码合并。                     |
| **`taxpayer_id`**                | VARCHAR      | **纳税人识别号**            | 财务开票和税务核实的核心标识。                               |
| **`business_term`**              | VARCHAR      | **营业期限**                | 用于校验企业是否在合法存续期内。                             |
| **`credit_code`**                | VARCHAR      | **统一社会信用代码**        | **身份唯一标识（极其重要）**。用于跨表去重和强一致性校验。   |
| **`status`**                     | VARCHAR      | **经营状态**                | 例如：存续、在业、吊销、注销。招投标库中通常只保留“存续/在业”。 |
| **`address`**                    | VARCHAR      | **完整注册地址**            | 营业执照上的原始完整地址。                                   |
| **`province / city / district`** | VARCHAR      | **省/市/区县**              | **智能解析地址**。通过地理位置解析器（Geo-Parser）从原始混乱文本中精准拆解，消除“市里带省”等脏数据，用于细粒度地域过滤。 |
| **`industry`**                   | VARCHAR      | **所属行业**                | 用于大类业务划分。                                           |
| **`insured_count`**              | VARCHAR      | **参保人数**                | 衡量企业实际运营规模和活跃度的重要侧面指标。                 |
| **`business_scope`**             | TEXT         | **经营范围**                | **业务匹配核心**。说明了这家公司能干什么。                   |
| **`metadata`**                   | JSON         | **全量信息合集 (RAG 核心)** | **大模型喂料核心**。不仅包含了所有非标准的野字段，也包含了当前行的所有有效标准字段。以 JSON 字典格式呈现，最适合大模型直接阅读和提取信息。 |

### 二、 法律法规库 (`law`)

**业务定位：** 招投标活动合规性审查、政策依据查询的基础知识库。

#### 1. 向量表：`milvus_law`

| **字段**                   | **类型** | **业务含义与作用**                                           |
| -------------------------- | -------- | ------------------------------------------------------------ |
| **`id`**                   | String   | 唯一主键 (如 `LAW_000001`)。                                 |
| **`core_text_for_bge_m3`** | Text     | **政策核心文本**。包含“标题 + 截断后的正文 + 全量 JSON 附加信息”。 |
| **`source`**               | String   | 来源网站。可用于限定只搜索某个特定政府网站发布的政策。       |
| **`pub_date`**             | String   | 发布时间。常用于 Milvus 前置标量过滤（例如：只查 2023 年以后的新规）。 |

#### 2. 关系表：`mysql_law`

| **字段**             | **类型** | **业务含义与作用**                                     |
| -------------------- | -------- | ------------------------------------------------------ |
| **`id` / `source`**  | VARCHAR  | 唯一主键与来源网站。                                   |
| **`title`**          | VARCHAR  | 法规标题（核心底线字段）。                             |
| **`pub_date`**       | DATE/STR | 发布时间。                                             |
| **`effective_date`** | DATE/STR | 生效时间。用于判断政策是否已经开始执行。               |
| **`content`**        | TEXT     | **完整正文**。保留未被截断的、原汁原味的超长法规全文。 |
| **`metadata`**       | JSON     | **全量信息合集**。用于 RAG 系统直接读取法规的全貌。    |

### 三、 产品/商品信息库 (`product`)

**业务定位：** 构建供应商商品库，用于评估报价合理性、寻找平替材料。

#### 1. 向量表：`milvus_product`

| **字段**                   | **类型** | **业务含义与作用**                                           |
| -------------------------- | -------- | ------------------------------------------------------------ |
| **`id` / `source`**        | String   | 唯一主键与来源。                                             |
| **`core_text_for_bge_m3`** | Text     | **商品描述文本**。拼接了商品名称和全量 JSON 详情，解决“同物不同名”检索问题。 |

#### 2. 关系表：`mysql_product`

高度结构化的商品台账。

| **字段**                                     | **类型** | **业务含义与作用**                                           |
| -------------------------------------------- | -------- | ------------------------------------------------------------ |
| **`id`、`source`**                           | VARCHAR  | 主键与数据来源。                                             |
| **`product_name`**                           | VARCHAR  | 商品名称（核心底线字段）。                                   |
| **`supplier`**                               | VARCHAR  | **供应商名称**。可通过此字段与 `mysql_company` 表联动查询资质。 |
| **`price`**                                  | DECIMAL  | **纯净数字报价**。已自动抹平“万元/元”差异，统一换算为“元”数字，支持大小比较检索。 |
| **`unit`**                                   | VARCHAR  | **物理单位**（如：吨、公斤、套）。大模型理解商品规格的核心锚点。 |
| **`supplier_address`**                       | VARCHAR  | 供应商原始地址。                                             |
| **`province / city / county`**               | VARCHAR  | **智能解析出的省市区**。用于做物流评估或本地化现货采购过滤。 |
| **`contact_person / contact_phone / email`** | VARCHAR  | 采购联系方式。                                               |
| **`product_params`**                         | TEXT     | 产品规格参数（长文本属性）。                                 |
| **`metadata`**                               | JSON     | **全量信息合集**。大模型读取商品参数、价格及隐含长尾属性的万能字典。 |

| **`purchaser`**            | String   | 甲方（采购人）。                                  |
